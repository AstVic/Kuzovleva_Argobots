# Отчет

## 1. Контекст диапазона изменений

- исправления механики кражи задач;
- расширение экспериментов для multi-Jacobi и single-runtime сценариев;
- развитие базового и cost-aware планировщиков;
- добавление/обновление результатов измерений и скриптов автоматизации.

## 2. Сводка изменений по проекту

Ключевые кластеры изменений:
1. **Планировщики** (`workstealing_scheduler/`):
   - доработка старого WS-планировщика (steal counters, кража половины пула, корректное помещение украденной работы);
   - существенная переработка нового cost-aware планировщика (sampled victim selection, критерии дисбаланса, fallback-режим кражи).
2. **Вычислительное приложение Jacobi-3D** (`jac3d_argobots/`):
   - интеграция счетчиков краж;
   - расширенный runtime для нескольких задач в одном процессе (`jac3d_multi_runtime.c`);
   - регистрация стоимостей задач перед их постановкой в пул.
3. **Экспериментальная инфраструктура** (`scripts/` и `scripts/results_scheduler_compare/`):
   - новые скрипты прогона single-runtime сценариев 4x и 8x;
   - нормализация набора текстовых отчетов по старому/новому планировщикам;
   - обновление `summary.csv` и добавление отдельных `summary_single_runtime_4x.csv`, `summary_single_runtime_8x.csv`.

## 3. Что именно исправлено и улучшено

### 3.1 Старый work-stealing планировщик

В старом планировщике (`abt_workstealing_scheduler.c`) закреплены следующие улучшения:
1. Введены потокобезопасные глобальные счетчики:
   - число операций кражи;
   - число украденных задач.
2. При пустом локальном пуле теперь выполняется попытка украсть не только одну задачу, а до половины задач из пула-жертвы (`victim_size/2`, минимум 1).
3. Украденные задачи размещаются в локальный пул вора (`ABT_self_schedule(thread, pools[0])`), что исправляет логическую ошибку прежних версий, где задача могла оставаться в неправильной очереди.

### 3.2 Новый cost-aware планировщик

Новый планировщик (`abt_workstealing_scheduler_cost_aware.c`) был переработан в сторону более реалистичной модели нагрузки:
1. Нагрузка пула разделена на:
   - `queued_*` (задачи в очереди);
   - `running_*` (задачи, уже исполняемые на ES).
2. Введена sampled-стратегия выбора жертвы:
   - из случайной подвыборки пулов (размер до 3);
   - с учетом порога минимальной кражи и коэффициента дисбаланса.
3. Добавлен fallback-механизм:
   - если cost-aware критерий не нашел жертву, используется более грубая эвристика «найти непустой пул и украсть до половины задач».
4. Добавлены публичные API для чтения/сброса статистики краж (`ws_get_steal_ops_count`, `ws_get_stolen_tasks_count`, ...).

## 4. Подробная модель "что такое одна ULT-задача"

В этом проекте одна ULT-задача — это один объект `ABT_thread`, создаваемый вызовом `ABT_thread_create(...)` для конкретного чанка вычислений:
- в `jac3d.c` ULT покрывает диапазон строк `i` (`start_i .. end_i`) либо для фазы A, либо для фазы B;
- в `jac3d_multi_runtime.c` ULT аналогично соответствует чанку конкретной задачи Jacobi и конкретной фазы;
- в редукции (`abt_reduction.c`) ULT соответствует куску массива входных значений, где выполняется локальная редукция максимума/другой операции.

Таким образом, «единица планирования» в новом планировщике — именно ULT на уровне chunk-подзадачи (а не вся итерация Jacobi целиком).

## 5. Как оценивается стоимость задач (cost model)

### 5.1 Базовый Jacobi (`jac3d.c`)

Перед постановкой ULT в пул вызывается `ws_push_task_estimate(pool_id, estimate)`.
Оценка здесь задается грубо как:

`rows_per_thread * L * L`

где `rows_per_thread = (L - 2) / num_threads`.

### 5.2 Multi-runtime Jacobi (`jac3d_multi_runtime.c`)

Используется более выразительная модель:
- `estimate_chunk_cost(size, rows, phase)`:
  - базово `rows * (size-2)^2`;
  - множитель фазы: `1.2` для фазы A, `1.0` для фазы B, `0.05` для reduction-фазы.
- `estimate_reduction_cost(len)` для редукционных задач.

То есть планировщик получает не только «размер чанка», но и поправку на характер фазы.

## 6. Полный разбор механизма кражи в новом планировщике

### 6.1 Структуры состояния

Для каждого пула ведутся:
- FIFO буфер точных оценок задач (`est_buffer`);
- суммарная ожидающая нагрузка (`queued_estimated`, `queued_count`);
- суммарная исполняемая нагрузка (`running_estimated`, `running_count`).

Это позволяет оценивать **полную текущую нагрузку** как:

`total_load = queued_estimated + running_estimated`.

### 6.2 Нормальный путь (локальная работа есть)

1. ES берет задачу из своего локального пула (локальный индекс 0).
2. Списывает оценку из FIFO `ws_pop_task_estimate(rank)`.
3. Оборачивает исполнение в `ws_execute_task_with_estimate(...)`, которая:
   - увеличивает `running_*` перед `ABT_self_schedule`;
   - уменьшает `running_*` после завершения задачи.

### 6.3 Путь кражи (локальный пул пуст)

#### Шаг A: поиск жертвы через sampled victim selection

Выбор производится функцией `ws_find_victim_sampled(...)`:
1. Берется до `WS_VICTIM_SAMPLE_SIZE=3` случайных кандидатов среди других пулов.
2. Для каждого кандидата читается `ws_get_pool_total_load(candidate)`.
3. Выбирается кандидат с максимальной load.
4. Кража разрешается только если выполняются условия:
   - `best_load > WS_MIN_STEAL_COST`;
   - `best_load > local_load * WS_LOAD_IMBALANCE_RATIO` (где ratio=1.15).

#### Шаг B: сколько красть

После выбора жертвы:
- целевая стоимость кражи `target_cost = 0.5 * (victim_load - local_load)`;
- нижняя граница `target_cost >= WS_MIN_STEAL_COST`.

То есть вор не пытается «обнулить» дисбаланс полностью, а забирает примерно половину разницы нагрузки.

#### Шаг C: цикл кражи

В цикле выполняется:
1. Поп одной ULT из пула жертвы.
2. Списание точной оценки у жертвы: `ws_pop_task_estimate(victim)`.
3. Выполнение украденной задачи на текущем ES.
4. Накопление `stolen_cost += est`.

Условия досрочного выхода:
- достигнута `target_cost`;
- у жертвы больше нет задач;
- украдена «дешевая» задача (`est <= WS_CHEAP_TASK_COST`, где 32.0), чтобы не тратить лишнее на агрессивную кражу мелких задач;
- после одной кражи обнаружена локальная задача у вора — тогда вор возвращается к локальной очереди.

#### Шаг D: учет статистики

Если украдена хотя бы одна задача, увеличиваются:
- `g_steal_operations` на 1;
- `g_stolen_tasks` на число фактически украденных ULT.

### 6.4 Fallback-путь (если sampled-логика не выбрала жертву)

Если дисбаланс не подтвердился по cost-aware критериям, включается fallback:
1. Ищется первый непустой удаленный пул.
2. Оценивается его размер.
3. Крадется до `victim_size/2` задач (минимум 1), как в baseline-эвристике.
4. Для каждой украденной задачи также списывается оценка и выполняется на текущем ES.

Это делает алгоритм более устойчивым: даже если модель стоимости «не уверена», система не блокируется в состоянии idle.

## 7. На что теперь влияет `threads`

`threads` напрямую задает степень разбиения задач (chunking granularity):
1. В классическом `jac3d.c`:
   - больше `threads` → меньше `rows_per_thread` → мельче ULT-задачи;
   - оценка одной ULT уменьшается (т.к. зависит от rows), но их число растет.
2. В `jac3d_multi_runtime.c`:
   - для каждой проблемы число чанков на фазу: `chunks = min(num_threads, interior_rows)`;
   - больше `threads` увеличивает потенциальный параллелизм и давление на планировщик;
   - одновременно растет overhead и вероятность мелких задач, для которых эвристика cheap-task cutoff (32.0) начинает чаще срабатывать.

Следствие для нового планировщика: при больших `threads` он чаще работает в режиме «коротких краж + возврат к локальным задачам», что снижает накладные расходы на чрезмерное воровство мелочи.

## 8. На что влияет `xstreams`

`xstreams` определяет число ES/пулов и ширину пространства кражи:
1. В обоих WS-планировщиках создается по одному пулу на ES.
2. Для каждого scheduler-а пулы задаются в **циклически сдвинутом порядке**, поэтому локальный пул для данного ES — это `pools[0]` его view, а удаленные пулы — `pools[1..]`.
3. В новом алгоритме рост `xstreams`:
   - увеличивает число потенциальных жертв;
   - но sampled-поиск ограничен максимум 3 кандидатами, поэтому стоимость выбора жертвы не растет линейно с `xstreams`.

Для `jac3d_multi_runtime.c` есть дополнительная особенность `task_pool_id(...)`: задачи намеренно раскладываются только по `active_pools = num_xstreams / 2` (минимум 1), провоцируя контролируемый дисбаланс и делая поведение кражи экспериментально заметным.

## 9. Экспериментальные и инфраструктурные изменения

1. Добавлен отдельный исполняемый multi-runtime сценария (`jac3d_multi_runtime.c` + бинарь `scripts/jac3d_multi_runtime`).
2. Добавлены отдельные кампании измерений `single_runtime_4x` и `single_runtime_8x`.
3. Обновлены агрегированные таблицы результатов (summary/benchmark) для сравнения старого и нового планировщиков на расширенном наборе конфигураций.

## 10. Вывод (академическая интерпретация)

В рассматриваемом диапазоне коммитов архитектура планирования эволюционирует:
- от простого event-driven work stealing с линейным просмотром пулов
- к гибридному cost-aware механизму, который сочетает:
  1) оценку текущей стоимости (queued + running),
  2) стохастический bounded-overhead выбор жертвы,
  3) частичное выравнивание дисбаланса по целевой стоимости,
  4) защитный fallback на size-based stealing.

Именно эта комбинация делает новый планировщик более практичным для неоднородных нагрузок (разные размеры задач, разные фазы вычислений, multi-problem runtime), где «чисто жадные» или «чисто исторические» эвристики оказываются нестабильными.
